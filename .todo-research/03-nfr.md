## 4.1 Licensing and Intellectual Property (LIPR)

| Code | Requirement (Romanian, verbatim) | Implementation implication |
|------|----------------------------------|----------------------------|
| LIPR 001 | CNAS urmează să furnizeze următoarele medii de operare a SI „Protecția Socială": Mediul de producție; Mediul de testare/instruire; Mediul de dezvoltare. Mediile de operare a SI „Protecția Socială" vor fi furnizate în cadrul platformei tehnologice guvernamentale comune Mcloud. | Provision three CI/CD-driven environments (dev, test/training, prod) on Mcloud-compatible infrastructure; codify environment provisioning as IaC (Terraform/Ansible) targeting Mcloud. |
| LIPR 002 | Furnizorul va livra fără careva plată suplimentară toate licențele necesare (dacă e cazul) implementării și operării SI „Protecția Socială" în cadrul celor 3 medii puse la dispoziție de CNAS (exemplu: sistem de operare, SGBD, software specific, biblioteci, soluții COTS). | Inventory and bundle every paid license (OS, RDBMS, libraries, COTS) into the supplier deliverable; prefer FOSS (PostgreSQL/Linux) to avoid recurring license cost across 3 environments. |
| LIPR 003 | Cantitatea licențelor oferite trebuie să permită accesarea și utilizarea SI „Protecția Socială" (în orice mediu în care funcționează) nu trebuie să limiteze numărul de utilizatori autorizați, precum și utilizatori anonimi și sisteme externe. Nu vor exista restricții cu privire la numărul de documente, tranzacții sau mod de accesare a SI „Protecția Socială" (exemplu: limitări la accesare concurentă). | Choose stack with unlimited-user licensing model (per-core or open-source); avoid per-seat/CCU-limited components such as Oracle Named User or proprietary BI tools. |
| LIPR 004 | Cantitatea licențelor oferite trebuie să permită accesarea API-urilor expuse de SI „Protecția Socială" de orice aplicație și sistem extern. | API-gateway tier must not impose per-client license caps; ensure no metered API products are used at infrastructure layer. |
| LIPR 005 | Furnizorul va transmite către CNAS toate drepturile asupra dezvoltărilor, ajustărilor, configurărilor și personalizărilor efectuate pentru implementarea SI „Protecția Socială" conform cerințelor. Acestea pot fi aferente produselor soft terțe licențiate, sau pot fi componente elaborate în cadrul proiectului. | All custom code, configurations and customizations stored in CNAS-owned Git repository; copyright assignment clauses included in contract. |
| LIPR 006 | Furnizorul va transmite dreptul de autor asupra întregului cod sursă elaborat în cadrul SI „Protecția Socială". | Hand-over agreement for full source-code copyright; ensure clean third-party dependency licenses (no GPL contamination of proprietary modules). |
| LIPR 007 | Orice date stocate în cadrul bazelor de date aferente SI „Protecția Socială" sunt proprietatea CNAS. Accesul la aceste date pe întreaga perioada de contractare a Furnizorului, cât și după, este subiect al cerințelor și clauzelor de confidențialitate a informației. | DPA/NDA in contract; supplier accounts on prod limited and audited; data export procedures documented for hand-over. |
| LIPR 008 | Furnizorul va prezenta modelul său de licențiere propus pentru SI „Protecția Socială" care trebuie să corespundă cerințelor LIPR 001 – LIPR 007. Furnizorul va descrie modelul de licențiere propus, argumentând de ce acesta este cel optim pentru CNAS și va prezenta o analiză comparativă cu alte modele de licențiere oferite de obicei pentru soluția propusă. | Deliver formal "Licensing Model" document in bid; comparative table vs. alternatives (e.g. PostgreSQL vs Oracle vs SQL Server). |

## 4.2 Architecture (ARH)

| Code | Requirement (Romanian, verbatim) | Implementation implication |
|------|----------------------------------|----------------------------|
| ARH 001 | Arhitectura SI „Protecția Socială" trebuie să fie bazată pe standarde deschise. | Restrict tech choices to those backed by ISO/IETF/W3C/OASIS specs (REST/OpenAPI, OAuth2, OIDC, SAML, XML/JSON-Schema). |
| ARH 002 | Arhitectura SI „Protecția Socială" trebuie să fie una orientată pe servicii (SOA). | Decompose system into services with explicit contracts; service registry/catalog; consider REST + AMQP/Kafka for inter-service messaging. |
| ARH 003 | Arhitectura SI „Protecția Socială" va fi concepută în manieră integrată, dezvoltată cu aplicarea celor mai bune practici în domeniu (exemplu: principii de arhitectură și arhitecturi de referință aliniate TOGAF 9.1). | Produce Solution Architecture Document using TOGAF 9.1 ADM artifacts (business, data, application, technology views). |
| ARH 004 | Arhitectura SI „Protecția Socială" va fi una multinivel de tip client-server, organizată în cel puțin 3 nivele (exemplu: nivelul de prezentare, nivelul aplicației și nivelul de date). | Enforce strict 3-tier separation: Presentation (SPA), Application (.NET/Java services), Data (RDBMS); use architecture tests to police layer boundaries. |
| ARH 005 | Arhitectura SI „Protecția Socială" trebuie să fie adaptată la implementarea și utilizarea în mediile virtualizate. | All services containerized (Docker) and orchestrable in VM/hypervisor; avoid bare-metal-only dependencies. |
| ARH 006 | Caracteristici ale unui sistem informațional cu arhitectură orientată pe implementare în medii virtualizate, sunt: conștient de latență, conștient de căderi de componente, paralelizabil, conștient de utilizarea resurselor. | Implement retries, circuit breakers (Polly/Resilience4j), bulkheads, health probes, horizontal partitioning, resource limits in container manifests. |
| ARH 007 | Comunicarea între toate componentele sistemului se va face securizat, utilizând în acest scop interfețele interne ale componentelor sistemului. | mTLS between services; internal API gateway; no plaintext intra-cluster traffic. |
| ARH 008 | Arhitectura tehnologică a SI „Protecția Socială" trebuie să aibă un nivel de toleranță ridicat la căderi și să nu conțină elemente SPOF. | Run >=2 replicas of every stateless service; clustered RDBMS (Patroni/Citus); redundant load balancers and storage. |
| ARH 009 | Arhitectura tehnologică a SI „Protecția Socială" trebuie să asigure utilizarea rațională și balansată a resurselor de procesare. | Configure Kubernetes HPA/VPA, resource requests/limits, balanced node pools, cost-aware autoscaling. |
| ARH 010 | În calitate de tehnologie de desfășurare și balansare a încărcării SI „Protecția Socială" este binevenită utilizarea Kubernetes. | Use Kubernetes (likely managed on Mcloud) as default orchestrator; Helm charts as deployment unit. |
| ARH 011 | Exploratorul Web va servi în calitate de aplicație client a utilizatorilor autorizați pentru accesarea funcțiilor business ale SI „Protecția Socială" accesul la care le este autorizat. | Browser-based SPA only; no thick clients. Verify on Chromium, Firefox, Edge. |
| ARH 012 | Nivelul Prezentare al arhitecturii SI „Protecția Socială" nu trebuie să implementeze reguli de business, cu excepția validării datelor de intrare. | Front-end limited to view + input validation; all business rules in Application services; reinforce via architecture tests. |
| ARH 013 | Nivelul Aplicație al arhitecturii SI „Protecția Socială" trebuie să fie independent de Nivelul Prezentare și sistemele informaționale care vor accesa direct Nivelul Aplicație (prin intermediul unor API specializate). | Application layer exposes versioned REST APIs; no direct entity binding; UI and external systems share the same contract. |
| ARH 014 | Nivelul Aplicație al arhitecturii SI „Protecția Socială" trebuie să fie dezvoltat în baza unor principii modulare bazat pe reutilizarea componentelor și interfețelor abstracte. | Use Clean/Hexagonal architecture; programmer-facing interfaces; module boundaries enforced via project boundaries. |
| ARH 015 | Componentele aferente Nivelului Aplicație al SI „Protecția Socială" trebuie să comunice între ele prin interfețe interne/funcții interne dedicate (cuplare strânsă). | Intra-application communication via DI-injected interfaces; no DB-shared state between modules. |
| ARH 016 | Componentele aferente Nivelului Aplicație al SI „Protecția Socială" trebuie să fie accesibile sistemelor informaționale externe doar prin API-uri dedicate acestor scopuri. | External access only via dedicated public/integration APIs (separate from internal interfaces); API gateway as single ingress. |
| ARH 017 | Nivelul logicii de business al SI „Protecția Socială" trebuie să asigure accesarea concurentă a obiectelor și funcțiilor SI „Protecția Socială". | Stateless services + optimistic concurrency (row-version/ETag); distributed locks (Redis/Postgres advisory) where required. |
| ARH 018 | Structura bazei de date SI „Protecția Socială" trebuie să corespundă următoarelor cerințe: standardizarea principiilor denumirii variabilelor și a câmpurilor bazei de date utilizându-se formatul PascalCase; utilizarea simbolului " " in toate variabilele, sau excluderea lui din toate denumirile câmpurilor bazei de date; utilizare unui singur limbaj (engleza) pentru descrierea câmpurilor tabelei bazei de date (este inacceptabilă utilizarea simultană a doua și mai multe limbaje, engleză și română de exemplu, pentru descrierea tabelelor sau câmpurilor bazei de date); utilizarea tipurilor de date ale limbajului XML (exemplu: pentru descrierea câmpului IDNO se va folosi STRING(13) nu CHAR(13)). | Adopt DB naming convention guide: PascalCase, English only, XML-Schema-compatible types (`VARCHAR(13)` not `CHAR(13)`); enforce via migration linter. |
| ARH 019 | SI „Protecția Socială" trebuie să suporte un model de date integrat pentru datele de referință. | Single shared `Reference` schema (nomenclators/classifiers) used by all modules; no copy-paste of code tables. |
| ARH 020 | SI „Protecția Socială" trebuie să susțină setul de caractere UTF-8 pentru datele stocate și să asigure ordonarea ascendentă (de la A la Z sau de la 0 la 9) și descendentă (de la Z la A și de la 9 la 0). Pentru tipurile de date calendaristică și timp crescător înseamnă că valorile evenimentelor anterioare le preced pe cele ulterioare (exemplu: 05/01/2019 va fi poziționat înainte de 05/01/2020). | DB cluster initialized with `UTF8` encoding + locale collation that handles RO/RU diacritics; ordering rules enforced in queries. |
| ARH 021 | Baza de date a SI „Protecția Socială" trebuie să asigure posibilitatea migrării și populării cu seturile de date deținute de CNAS. | Ship ETL/migration scripts and reusable loader services; document source-to-target mapping for every legacy dataset. |
| ARH 022 | SI „Protecția Socială" trebuie să asigure stocarea datelor textuale în baza de date în cel puțin 2 versiuni lingvistice: română și rusă. | Localized columns (`NameRo`, `NameRu`) or translations table; UI fallback chain. |
| ARH 023 | Datele stocate și gestionate prin intermediu SI „Protecția Socială" trebuie să fie accesibile doar prin intermediul componentelor funcționale furnizate de Nivelul Aplicație a arhitecturii sistemului informațional. | No direct DB access for clients; DB users restricted to app-service accounts; ad-hoc reporting goes through Application APIs or a read-replica with row-level security. |
| ARH 024 | Nivelul de Date al SI „Protecția Socială" trebuie să fie neutru și independente în raport cu Nivelul Aplicație al arhitecturii SI „Protecția Socială". | Use vendor-agnostic ORM (EF Core/Hibernate); avoid proprietary stored procedures for business logic. |
| ARH 025 | Nivelul de Date al arhitecturii SI „Protecția Socială" trebuie să fie optimizat în așa fel, încât să asigure accesul rapid la date pentru efectuarea tranzacțiilor și generarea rapoartelor statistice. Generarea rapoartelor statistice trebuie să nu afecteze performanța operațiilor tranzacționale ale SI „Protecția Socială". | OLTP vs OLAP separation: read-replica or data warehouse (CDC via Debezium) for reporting workloads. |
| ARH 026 | Modelul de date implementat trebuie să fie documentat în detaliu. Documentarea trebuie să conțină atât descrierea tehnică a nivelului de date (exemplu: XSD), cât și descrierea semantică (asocierea structurii datelor la entitate și proprietățile acesteia). Descrierea semantică a datelor trebuie să fie disponibilă utilizatorilor în cadrul SI „Protecția Socială" (exemplu: pentru procesele de configurare a rapoartelor). | Auto-generate data dictionary (XSD + semantic glossary) from schema metadata; expose searchable inside admin UI. |
| ARH 027 | Înregistrările aferente obiectelor informaționale ale bazei de date trebuie să conțină un identificator unic atribuit automat la nivel de sistem informațional. Algoritmul de atribuire a identificatorului unicat trebuie să poată fi configurat prin intermediul mijloacelor SI „Protecția Socială" și să asigure facilități de verificare a admisibilității valorii identificatorului. | BIGINT identity PKs internally + Sqids encoding at API boundary; configurable ID-generation strategy per entity (sequence/UUIDv7/custom). |
| ARH 028 | Arhitectura de date a SI „Protecția Socială" trebuie să asigure integritatea, consistența și veridicitatea datelor în cazul accesării simultane a acestora de diferite entități (exemplu: utilizatori autorizați, procese interne, aplicații externe etc.). | Transactional boundaries clearly defined; row-versioning (`xmin`/`rowversion`); idempotent endpoint design. |
| ARH 029 | Cerințe pentru proiectarea bazei de date: - Baza de date trebuie să aibă elemente de restricționare a integrității; - Baza de date trebuie să aibă cel puțin gradul 3 de normalizare; - Relațiile multe-la-multe (many-to-many) nu trebuie aplicate entităților principale ale bazei de date. | FK/CHECK/UNIQUE constraints mandatory; 3NF baseline; M2M only via dedicated junction tables on secondary entities. |

## 4.3 Technology Stack (TS)

| Code | Requirement (Romanian, verbatim) | Implementation implication |
|------|----------------------------------|----------------------------|
| TS 001 | Tehnologiile propuse de Furnizor pentru dezvoltarea SI „Protecția Socială" trebuie să fie accesibile pentru cel puțin 3 companii specializate în dezvoltarea soluțiilor software care activează pe piața locală a Republicii Moldova | Use mainstream stacks (e.g. .NET, Java/Spring, Node, React/Angular) for which >=3 Moldovan vendors can take over; avoid niche frameworks. |
| TS 002 | Componentele SI „Protecția Socială" trebuie să fie independente în raport cu platforma tehnologică pe care sunt rulate (cu excepția cazurilor în care astfel de cerințe rezultă în mod explicit din aceste Specificații Tehnice). | Containerize everything; avoid OS-specific APIs; runtime-portable code (.NET 8 + Linux, JVM). |
| TS 003 | SI „Protecția Socială" trebuie să poată fi desfășurat în medii virtualizate (SI „Protecția Socială" trebuie să fie conform cerințelor de desfășurare a sistemelor informaționale pe platforma guvernamentală tehnologică comună Mcloud). | Comply with Mcloud onboarding checklist (image hardening, audit, scanning). |
| TS 004 | SI „Protecția Socială" va utiliza standarde deschise pentru formate și protocoale de comunicare. | Use JSON/XML/CSV/PDF/A, REST/OpenAPI/SOAP/MQTT only; no proprietary RPC. |
| TS 005 | Tehnologiile stivei tehnologice a SI „Protecția Socială" trebuie să fie omogene (număr minim de tehnologii diferite, adică spre exemplu aceleași sisteme de operare pentru serverele middleware și serverele bază de date). | Standardize on one Linux distro (e.g. RHEL/Ubuntu LTS) for all servers; one RDBMS engine; one main language. |
| TS 006 | Serviciile expuse publicului de către SI „Protecția Socială" trebuie să fie neutre din punct de vedere tehnologic (exemplu: sistem de operare, navigator Web etc.). | Web standards (HTML5/ES2020); SPAs avoid OS/browser-specific APIs; APIs language-agnostic. |
| TS 007 | Software-ul generic recomandat pentru exploatarea componentelor SI „Protecția Socială" este navigatorul Web. | All admin/user features delivered as browser apps; minimal CLI for ops only. |
| TS 008 | SI „Protecția Socială" trebuie să asigure setarea, modificarea, procesarea, stocarea și accesarea datelor textuale în format unicod (exemplu: folosind standardul de codificare UTF-8). | Set UTF-8 in DB/API/UI; all collations Unicode-aware. |
| TS 009 | SI „Protecția Socială" trebuie să fie accesibil pentru orice utilizator conectat la Internet prin intermediul unui dispozitiv standard (exemplu: PC, notebook, tabletă, smartphone etc.). | Responsive design; no bespoke device dependencies. |
| TS 010 | Pentru dezvoltarea și operarea SI „Protecția Socială" este binevenită utilizarea platformelor FOSS (free open source software) portabile pe sistemele de operare UNIX și WINDOWS. | Default to FOSS (.NET on Linux/Win, Node/JVM, PostgreSQL, NGINX). |
| TS 011 | Componentele Nivelului Aplicație al arhitecturii SI „Protecția Socială" trebuie să fie dezvoltate utilizându-se limbaje de programare/framework-uri utilizate pe larg de industria TIC a Republicii Moldova (exemplu: C#, Java, PHP, ASP.NET Core, Spring Framework, Laravel, AngularJS, React, VueJS, NodeJS etc.) și accesibile specialiștilor TIC din cadrul CNAS. | Backend: ASP.NET Core / Spring; Frontend: React/Angular/Vue. |
| TS 012 | Toate componentele funcționale al SI „Protecția Socială" (exemplu: sistem de operare, middleware, SGBD etc.) trebuie să dispună de capacitate de a fi desfășurate și opera în medii virtualizate. | Verify each chosen product has supported VM/container deployment. |
| TS 013 | Furnizorul va include în oferta sa tehnică specificațiile detaliate ale stivei tehnologice propuse (în limitele alternativelor/opțiunilor disponibile), având în vedere nevoile CNAS definite în prezenta Specificație Tehnică. | Deliver "Technology Stack Specification" document; map each requirement to product/version. |
| TS 014 | În calitate de SGBD pentru implementarea SI „Protecția Socială" sunt preferabile soluțiile open-source (de ex. PostgreSQL). | Use PostgreSQL 16+ as RDBMS; configure Patroni/Citus for HA. |
| TS 015 | În cazul în care platforma software/SGBD destinată dezvoltării și funcționării SI „Protecția Socială" se bazează pe soluții informatice comerciale care necesită procurarea licențelor, Furnizorul va include în oferta de preț costul tuturor licențelor necesare dezvoltării și funcționării SI „Protecția Socială" (Furnizorul trebuie să achiziționeze în numele CNAS totalitatea licențelor necesare dezvoltării și funcționării sistemului informațional). | Document full BoM including any commercial licenses; supplier procures on behalf of CNAS. |
| TS 016 | În cazul în care platforma software/SGBD destinată dezvoltării și exploatării SI „Protecția Socială" se bazează pe soluții informatice comerciale ce necesită licențiere, Furnizorul va include în oferta de preț delta costurilor privind licențierea în cazul: dublării numărului de utilizatori; dublării numărului de unități de procesare (CPU sau nuclee CPU); dublării numărului de noduri de server aplicație/bază de date. | Provide cost-delta tables for 2x scale across user/CPU/node axes; reinforces preference for unlimited-licensed FOSS. |
| TS 017 | SI „Protecția Socială" va opera în rețele TCP/IP și în special HTTPS. | Force HTTPS-only; HSTS; redirect HTTP->HTTPS; TLS 1.2+; modern cipher suites. |

## 4.5 Performance and Scalability (PSR)

| Code | Requirement (Romanian, verbatim) | Implementation implication |
|------|----------------------------------|----------------------------|
| PSR 001 | Timpul de răspuns al SI „Protecția Socială" la interogările parvenite din partea utilizatorilor/sistemelor informaționale externe nu trebuie să depășească: 1 secundă pentru execuția a 90% din interogări; 3 secunde pentru execuția a 99% din interogări; 5 secunde pentru generarea a 95% din rapoarte; 3 secunde pentru execuția a 90% din activitățile de management a documentelor. | Define SLOs (p90<=1s, p99<=3s; reports p95<=5s; doc ops p90<=3s); enforce via k6/JMeter perf tests in CI; alerting in Prometheus/Grafana. |
| PSR 002 | SI „Protecția Socială" trebuie să fie capabil să asigure funcționalitatea a cel puțin 1500 utilizatori autorizați și 500 utilizatori anonimi. | Plan for 2000+ user capacity; size cluster accordingly; load test at 2x baseline. |
| PSR 003 | SI „Protecția Socială" trebuie să asigure funcționarea în bune condiții în cazul a cel puțin 2000 sesiuni concurente (conexiuni ale utilizatorilor autorizați și sisteme informaționale externe). | Sticky-session-less design; DB connection pool sized for 2000 concurrent sessions; pgbouncer/proxysql. |
| PSR 004 | Orele de vârf ale exploatării sistemului informațional sunt următoarele pe parcursul orelor regulamentare de lucru ale autorităților publice din Republica Moldova (de la ora 08:00 până la ora 18:00). | Schedule backups/heavy jobs outside 08:00-18:00; scale-out during business hours via HPA schedule. |
| PSR 005 | Furnizorul trebuie să includă în ghidurile de administrare și operare a SI „Protecția Socială" descrierea proceselor care pot afecta/diminua performanța de funcționare a SI „Protecția Socială" și recomandările sale privind rularea concomitentă a acelor procese (exemplu: nu este recomandat să rulați procesul X pentru generarea de rapoarte zilnice concomitent cu procesul Y pentru generarea de copii de rezervă). | Document "Performance Operations Guide" with concurrency caveats per heavy process. |
| PSR 006 | Generarea KPI/rapoartelor statistice și accesarea datelor în scopul procesării și analizei acestora nu trebuie să afecteze performanța de lucru a SI „Protecția Socială" la nivelul procesării tranzacțiilor. | Route reporting to read-replica/warehouse; query timeouts; resource governance. |
| PSR 007 | Documentația SI „Protecția Socială" trebuie să specifice explicit rapoartele statistice sau proceselor cu impact semnificativ asupra performanței operării SI „Protecția Socială" și va conține recomandări ale Furnizorului cu privire la generarea rapoartelor respective, astfel încât indicatorii de performanță a SI „Protecția Socială" să nu fie influențați. | Catalogue "expensive reports" in admin docs with cost & scheduling guidance. |
| PSR 008 | Furnizorul trebuie să includă în oferta sa valorile minime garantate pentru indicatorii de performanță a SI „Protecția Socială", inclusiv platforma tehnologică recomandată pentru asigurarea indicatorilor de performanță așteptați. | Publish guaranteed perf KPIs + recommended HW sizing in proposal. |
| PSR 009 | SI „Protecția Socială" trebuie să fie capabil să proceseze zilnic cel puțin 300 000 tranzacții. | Capacity-test for >=300k tx/day (~3.5 tx/s average, much higher peak); right-size DB IOPS. |
| PSR 010 | Timpul de răspuns la o interogare tranzacțională a utilizatorului/sistemului informațional extern nu trebuie să depășească 1 secundă pentru interogările simple și 3 secunde pentru interogările complexe (cu excepția cazurilor generării rapoartelor statistice complexe). | Simple-query p100 <= 1s; complex-query p100 <= 3s; index strategy + EXPLAIN analyze in CI. |
| PSR 011 | SI „Protecția Socială" trebuie să dispună de capacități de a procesa tranzacții atât în timp real, cât și offline (procesare în fundal). | Provide background job framework (Hangfire/Quartz) + sync APIs; idempotent job design. |
| PSR 012 | SI „Protecția Socială" trebuie să permită creșterea capacității de procesare fără a întrerupe funcționarea sa. În acest scop, sistemul informațional trebuie să suporte extinderea orizontală a capacității de procesare (exemplu: adăugarea de noi instanțe server și balansarea încărcării). | Stateless services + rolling deploys; K8s HPA; zero-downtime DB scaling (read replicas, partitioning). |
