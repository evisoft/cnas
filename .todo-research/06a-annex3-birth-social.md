# Annex 3.1 (Birth event) & Annex 3.2 (Social Security) — Service-by-Service Findings

Source: `tor/TOR.md` lines 11206–14159 (PDF pages 168–225 of the Romanian TOR for SI "Protecția Socială").

## Cross-cutting business rules (apply to every service below)

These rules are repeated almost verbatim in every sub-section and are the implicit baseline for every workflow:

- For each insured person (Persoană asigurată) the system creates an electronic dossier (dosar electronic).
- For each request type the system stores template artifacts: cerere (application form), document rezultativ (fișa de calcul / decizie / contract), recipisă (receipt).
- For each request type the system maintains a classifier listing the mandatory attached documents (NOT applied to proactive services where no application is submitted).
- For each request type the system maintains a configurable waiting period for missing documents (perioada de așteptare a documentelor lipsă). When it expires, the system notifies the CTAS specialist.
- Benefits are either fixed-amount or computed from multiple factors (contributions calculated/paid, length of service, etc.).
- The system journals every event of the process (launch date/time, successful records, error records, completion date/time).
- Three registries are referenced everywhere:
  - Registrul persoanelor asigurate (Annex section 8.2.4)
  - Registrul deciziilor (Annex section 8.3.9)
  - Registrul de conturi de plată a prestațiilor (Annex section 8.3.10)
- The decision PDF is generated by the system, signed with electronic signature by the CNAS user, then forwarded for approval to the Șef CNAS (and in international-agreement / pension cases also via Șeful direcției). If returned for re-examination the user can edit the fișa de calcul and regenerate the decision.
- The calculated benefit amount is split by expenditure articles (articole de cheltuieli) per a CNAS-supplied classifier.
- Notifications are emitted at each major step to System, Beneficiar, Utilizator CNAS, Șef Direcție (where present), and Șef CNAS.
- Channels for citizen-facing intake: (a) online self-service after authentication (MPass implied), or (b) physical presentation at CNAS counter — the CNAS user files the request on behalf of the citizen; the printed cerere is signed hand-written ("olograf"), scanned, and attached. Recipisă is always produced.

### Service: 3.1-A — Indemnizația unică la nașterea copilului — One-time birth allowance

**Service code:** 3.1-A
**Provisioning modes:** Two modes:
  1. **Proactive** — automatic, based on data pulled from RSP (State Registry of Population) via MConnect.
  2. **On request** — application submitted online or at CNAS counter.

**Eligibility rules:** Mother of newborn child (Beneficiar = mother). System checks whether the same allowance was already granted for the same child (anti-duplication). Implicit eligibility from RSP record of birth.
**Required documents / data (proactive mode):** None — system pulls birth list from RSP and beneficiary data from RSP via MConnect. Beneficiary's CPAS code is assigned if not already present in the Registrul persoanelor asigurate.
**Required documents / data (on-request mode):** Cererea privind acordarea prestației + obligatory documents per classifier; recipisă is generated.

**External systems consulted:**
- **MConnect** (government interoperability platform) — gateway.
- **RSP** (Registrul de Stat al Populației) — source of newborn list and beneficiary data.

**Calculation / amount rules:** Fixed amount (sumă fixă) — calculation is system-driven via fișa de calcul, pre-filled with data available in system (beneficiary, periods considered). User may complete missing data with automatic recalculation. Amount divided by expenditure articles.

**Process steps (proactive — codes 3.1.1-A01 to 3.1.1-A04):**
1. **3.1.1-A01 (System)** — Lansarea procedurii de selectare a copiilor noi născuți. System launches the procedure (per scheduler, once a day, configurable; can be triggered by admin). Extracts list of newborns. If empty → process ends.
2. **3.1.1-A02 (System)** — Procesarea listei cu copiii noi născuți. For each record: validate and identify beneficiary (mother) by IDNP; if not in Registrul persoanelor asigurate, fetch from RSP and create new entry with auto-assigned CPAS code; generate Cerere and assign it to territorial CTAS by beneficiary's residence visa; emit notifications to CNAS user.
3. **3.1.1-A03 (Utilizator CNAS)** — Examinarea cererii și deciziei. User opens application; system generates fișa de calcul pre-completed with data and computed amount; user may amend with auto-recalculation; system warns if this allowance type was previously granted for the same child; user generates Decizia de acordare or Decizia de refuz (with reason); signs electronically; forwards to Șef CNAS.
4. **3.1.1-A04 (Șef CNAS)** — Approval. If no objection → e-signs, decision stored in Registrul deciziilor and process ends. Otherwise → returned to user (step A03).

**Process steps (on-request — codes 3.1.2-A01 to 3.1.2-A03):**
1. **3.1.2-A01 (Beneficiar/Utilizator CNAS)** — Depunerea cererii. Beneficiary authenticates, selects allowance, fills cerere, attaches mandatory documents (per classifier), signs, submits. Counter path: user files on behalf; printed cerere signed olograf, scanned, attached. Recipisă generated.
2. **3.1.2-A02 (Utilizator CNAS)** — Examinarea cererii și deciziei. User signs recipisă, validates per rules for this allowance type; generates fișa de calcul (pre-filled); generates Decizia (acordare or refuz, PDF), e-signs, forwards to Șef CTAS.
3. **3.1.2-A03 (Șef CNAS)** — Approval (same as proactive A04).

**Decision points:** Approve / return for re-examination by Șef CNAS. Anti-duplication check (same allowance for same child).
**Status transitions:** Cerere → în examinare → decizie generată → semnată Utilizator → aprobată Șef CNAS → stocată în Registrul deciziilor → procesată.
**Diagram present:**
- Proactive: `images/pdf_p173_full.png` (Figura A3.1.1) — clips `pdf_p173_1.png`, `pdf_p173_2.png`.
- On-request: `images/pdf_p176_full.png` (Figura A3.1.2) — clips `pdf_p176_1.png`, `pdf_p176_2.png`, `pdf_p176_3.png`.
**Output document / decision template:** Decizia privind acordarea prestației (PDF) OR Decizia privind refuzul acordării (with reason). Plus Notificare. Plus (on-request) Recipisă.
**Notifications:** To Beneficiar, Utilizator CNAS, Șef CNAS — at each step (notifications in system).
**Payment integration:** Not explicitly named here, but feeds into Registrul de conturi de plată a prestațiilor (section 8.3.10) — payout downstream.
**Edge cases / exceptions:** Validation failure on individual records → record skipped, error logged, processing continues to next. Beneficiary not in Registrul persoanelor asigurate → fetched from RSP, new CPAS code assigned. Previously granted for same child → warning to user. Re-examination loop on return from Șef CNAS.

### Service: 3.1-C — Indemnizație lunară pentru creșterea copilului, conform opțiunilor (persoanelor asigurate) — Monthly childcare allowance, by option (insured persons)

**Service code:** 3.1-C
**Reference:** Process identical to 3.1-A.
**Eligibility rules:** Insured persons; benefit "by option" — the insured chooses the duration option (e.g., 2 or 3 years).
**Special rule (Option-change re-examination):** If the applicant submits a new cerere that changes the initial option (e.g., originally granted for 3 years, now requests for 2 years), the system generates a NEW decision establishing the new amount AND a recalculation for prior periods. The previous decision in Registrul deciziilor is set to **statutul "Terminat"** with fields populated:
  - Motivul terminării
  - Data terminării
  - Numărul deciziei terminării
The termination is applied AFTER approval of the new decision.
**Status transitions:** Activ → Terminat (on option change, after new decision approved).
**Process steps:** Same as 3.1-A, plus the option-change recalculation/termination handling described above.
**Diagram present:** Inherits 3.1-A diagrams.
**Edge cases:** Option-change recalculation explicitly handled. Recalculation generates restitution/adjustment values for prior periods.

### Service: 3.2-A — Pensie pentru limită de vârstă — Old-age pension

**Service code:** 3.2-A
**Provisioning mode:** On request (la solicitare).
**Actors:** Sistem, Beneficiar, Utilizator CNAS, Șef CNAS.
**Eligibility rules:** Per Legea nr.156/1998 (age, contribution period — stagiu de cotizare per HG nr.165/2017). Pre-filled in fișa de calcul.
**Required documents / data:** Cerere privind acordarea pensiei + mandatory attached documents per classifier.
**External systems consulted:** Not explicitly enumerated in this sub-section, but Registrul persoanelor asigurate is consulted; data from contributions is internal.
**Calculation / amount rules:** Computed from contributions, periods of contribution, length of service; pre-filled fișa de calcul; user can amend with auto-recalc; result split by expenditure articles per CNAS classifier. **Critical comparison rule:** If newly-calculated pension/benefit amount is **less than that of a previous Decizie**, the user may generate a Decizia de refuz citing the reason; user may then additionally issue a Decizia de acordare for another type of benefit (with its own fișa de calcul).
**Process steps (codes 3.2-A01 to 3.2-A04):**
1. **3.2-A01 (Beneficiar/Utilizator CNAS)** — Depunerea cererii (online or counter). Recipisă generated.
2. **3.2-A02 (Utilizator CNAS)** — Examinarea cererii și deciziei. Sign recipisă; review. **30-day missing-documents hold**: if documents missing, cerere is held for 30 zile (perioada de așteptare, configurable per service type). System lets beneficiary/user attach missing documents only within this period; on expiry → notifies Specialist CTAS. Validate; generate fișa de calcul; system warns if any other or same pension type was previously granted for this Beneficiar; generate Decizia (acordare/refuz/alternate-grant, PDF); e-sign; forward to Șef CNAS.
3. **3.2-A03 (Beneficiar/Utilizator CNAS)** — Completarea cererii cu documentele lipsă. Beneficiary or CNAS user attaches missing documents listed in recipisă; system journals event.
4. **3.2-A04 (Șef CNAS)** — Approval. If OK → e-sign and process complete. Otherwise → return to A02. **Cascade rule:** when the new Decizia differs from the previous Decizia (e.g., amount increase, or different pension/benefit type), Registrul deciziilor is updated: the previous Decizia's period-of-grant gets its end-date set to the start-date of the new Decizia and its status changes to **"Terminat"**. Example explicitly given: beneficiary receives benefit type 1, requests type 2; if type 2 is granted, type 1's calculations and payments terminate.
**Decision points:** Approve / return for re-examination. Comparison with prior decision triggers alternate-grant or termination. Anti-duplication warning. 30-day document-wait expiry.
**Status transitions in Registrul deciziilor:** Activ → (on new decision granted for different type/amount) → Terminat (with end date set to new decision start date).
**Diagram present:** `images/pdf_p191_full.png` (Figura A3.2) — clips `pdf_p191_1.png`, `pdf_p191_2.png`, `pdf_p191_3.png`. Includes branch "Lipsesc documentele solicitate? Da → Completarea cererii cu documentele lipsă; Nu → continuă".
**Output document:** Decizia (acordare or refuz, plus optional alternate-grant Decizia) + Notificare + Recipisă.
**Notifications:** To Beneficiar, Utilizator CNAS, Șef CNAS at each step.
**Payment integration:** Registrul de conturi de plată a prestațiilor (8.3.10) handles downstream payment.
**Edge cases / exceptions:**
  - Missing documents → 30-day hold, configurable.
  - New calculated amount < previous decision amount → refuz; allow alternate-grant.
  - Previously granted same/other pension type → warning, cascade termination on new approval.
  - Re-examination loop.

## Cross-cutting workflow primitives identified

These appear repeatedly and should be modelled as **reusable workflow components**:

1. **OnRequestApplicationFlow** — 3-step pattern: Depunere → Examinare+Decizie (Utilizator CNAS) → Aprobare (Șef CNAS). With re-examination loop.
2. **OnRequestApplicationFlowWithDirectorate** — 4-step with intermediate Șef Direcției approval. Used for international-agreement services and 3.1-F.
3. **ProactiveBatchFlow** — daily scheduler → list extraction (RSP/SFS/SI PS) → record-by-record validation/identification → auto-creation in Registrul persoanelor asigurate (auto-CPAS code) → auto-Cerere generation → routing to territorial CTAS → user examination → decision approval.
4. **MissingDocumentsHold** — configurable per-service waiting period; user/beneficiary attaches documents within window; expiry triggers Specialist CTAS notification.
5. **DuplicateCheck** — system warns if same allowance type was previously granted for the same beneficiary/child/period.
6. **CascadeTermination** — when a new Decizia differs from a previous one (different type, higher amount, etc.), the previous Decizia's grant period is closed and its status set to **"Terminat"** (with Motivul terminării, Data terminării, Numărul deciziei terminării).
7. **PaperVoucherIssuance** — 3.2-AB / 3.2-AC pattern: voucher allocation → issuance → return-stub registration with attached scan.
8. **StockManagementAndDistribution** — 3.2-AD pattern: voucher stock + reservation/blocking + acts de predare-primire + per-CTAS/CNSM distribution.
9. **ContractAndAcordLifecycle** — 3.2-AD: contract (first time) → acord adițional (subsequent) → rescission via Acord de reziliere with factura fiscală de retur and voucher return.
10. **ReportingAndDeviation** — 3.2-AD usage reporting: 3 forms (1-BTS, 2-BTS, +third) per reporter type (Beneficiar / CNSM / Sanatoriu); deviations trigger Decizie de restituire a sumelor.
11. **CertificateGeneration** — 3.2-AE/AF/AG/AH/AI pattern: request → system generates from data → Șef CNAS signs → places in cabinetul personal.
12. **CabinetulPersonalAlBeneficiarului** — personal cabinet for citizens receiving signed output documents.

## Key implementation observations for TODO planning

1. **The "Indemnizația unică la nașterea copilului" (3.1-A) is the canonical template** — most birth-event services inherit from it (3.1-B, 3.1-C, 3.1-G, 3.2-AA).
2. **The "Indemnizație paternală" (3.1-D) is the no-proactive-mode template** — adopted by 3.1-H, 3.1-I, 3.1-J.
3. **The "Pensie pentru limită de vârstă" (3.2-A) is the canonical pension template** — inherited by all standard pensions 3.2-B through 3.2-J and 3.2-L.
4. **The "Alocația lunară de stat pentru veteranii de război" (3.2-M) is the canonical allowance template** — inherited by 3.2-N through 3.2-Y (most non-pension allowances).
5. **3.1-F and 3.2-K are the international-agreement templates** with three-level approval (Utilizator → Șef Direcție → Șef CNAS).
6. **3.2-S adds geographic eligibility logic** (CUATM classifier for left-bank Nistru) with automatic termination on residence change.
7. **3.2-U adds external-system (SFS) integration** for proactive intake; 3.2-Z adds internal-SI-PS proactive intake.
8. **3.2-AB / 3.2-AC are the paper-voucher templates** with return-stub registration and per-year duplicate check.
9. **3.2-AD is a fully separate 4-sub-process workflow** with stock management, CNSM external actor via MPass, contract/acord/factura fiscală lifecycle, multi-reporter usage reporting (3 forms), and contract rescission with factura fiscală de retur.
10. **3.2-AE/AF/AG/AH/AI are certificate-style services** with no fișa de calcul; output goes to Cabinetul personal al Beneficiarului.
11. **MPass authentication** explicitly required for external actors (CNSM) — implicitly assumed for citizen self-service portals.
12. **MConnect → RSP** is the primary data-pull mechanism for proactive birth-event services.
13. **MConnect → SFS** is the data-pull mechanism for 3.2-U.
14. **Payment integration via Trezorerie / banks / MPay** is NOT explicitly named in this range; payments are abstracted via Registrul de conturi de plată a prestațiilor (section 8.3.10). Concrete payment-channel integration is described elsewhere in the TOR (Annex 8.3.10 and beyond — out of scope of this read).
15. **Comparison-with-prior-decision logic** (3.2-A02 paragraph) is a critical rule: if newly-calculated amount < previous, allow Decizia de refuz with reason and optional alternate-grant.
16. **Cascade termination of prior decisions** (3.1-C, 3.2-A04) is a major data-integrity behaviour requiring atomic update of Registrul deciziilor.
17. **Configurable per-service parameters**: missing-documents waiting period (in days), expenditure-article classifier, mandatory-documents classifier, application/decision/recipisă templates — all stored as service-type metadata.
18. **The 30-day hold for old-age pension (3.2-A02) is explicitly 30 days**; for 3.2-M and others, the period is configurable in days.
